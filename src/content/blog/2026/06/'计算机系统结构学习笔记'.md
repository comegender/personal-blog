### 计算机系统结构的定义
+ 程序员所看到的计算机属性，即概念性结构与功能行为。对计算机系统而言，是指那些由程序员可见的系统属性
  + 所指程序员：汇编语言、机器语言、编译程序、操作系统
  + 程序员所看到的是为了程序能在机器上正确运行所必须了解的内容
+ 计算机系统的软、硬件界面，即机器语言机器语言程序员所看到的传统机器级所具有的属性
  + 计算机系统结构主要研究对软硬件界面的确定和软硬件功能分配

### 透明性
本来存在的事物或属性，从某种角度看似乎不存在
+ **浮点数表示、乘法指令**：对高级语言程序员、应用程序员透明，对汇编语言程序员、机器语言程序员不透明
+ **数据总线宽度、微程序**：对汇编语言程序员、机器语言程序员透明，对硬件设计者、计算机维修人员不透明

### 定量原理
1. 大概率事件优先原理
   + 对于大概率事件，赋予它优先的处理权和资源使用权，以获得全局的优化效果。
2. Amdahl定律
   + 加快某部件执行速度所能获得的系统加速比，受限于该部件的执行时间占系统中总执行时间的百分比
    $$加速比=\frac{系统性能_{改进后}}{系统性能_{改进前}}=\frac{总执行时间_{改进前}}{总执行时间_{改进后}}$$
   + 可改进比例$F_e$：在改进前的系统中，可改进部分的执行时间在总执行时间总的占比
   + 部件加速比$S_e$：可改进部分改进以后性能提高的倍数。它是可改进部分在改进前所需时间于改进后执行时间的比
   + 改进后程序总执行时间$T_n=T_0(1-F_e+\frac{F_e}{S_e})$
   + 系统加速比$S_n=\frac{T_0}{T_n}=\frac{1}{1-F_e+\frac{F_e}{S_e}}$

### 流水线的特点
+ 流水线将一个处理过程分解成若干个子过程，每个子过程由一个专门的功能部件来实现
+ 流水线中各段的时间应尽可能相等，否则将引起流水线堵塞、断流。时间长的段将成为流水线的瓶颈。
+ 效率
  + 连续不断的使用同一种功能
  + 通过时间：第一个任务从进入到流出
  + 排空时间：最后一个任务从进入到流出

### 流水线的分类
+ 按等级
  + 部件级：把处理机中的部件分段相互连接，是运算操作按流水方式处理
  + 处理机级：把指令执行过程分解为若干子进程，每个在独立的功能部件中执行
  + 系统级：把多个处理机串行连接，对同一数据流进行处理，每个完成任务中的一部分，前一台的结果放入存储器，作为后一台的输入
  ![alt text](/system-struct/image-15.png)
+ 按完成功能的多倍性
  + 单功能：流水线之间各段的连接关系固定不变，只能完成一种功能
  + 多功能：段之间的连接可以变化，不同的连接方式可以完成不同的功能
    + 静态：同一时间内，多功能流水线中的各段只能按照同一种功能的连接方式工作
    + 动态：在同一时间内，多功能流水线中的各段可以按照不同的方式连接，同时执行多种功能
    ![alt text](/system-struct/image-16.png)
+ 按是否有反馈回路
  + 线性流水线：流水线的各段串行连接，没有反馈回路，数据通过流水线中的各段时每一个段最多只流过一次
  + 非线性流水线：流水线中除了串行的连接外还有反馈回路
  ![alt text](/system-struct/image-17.png)
+ 根据任务流入和流出的顺序是否相同
  + 顺序流水线：流水线输出端任务的流出顺序与输入端的任务的流入顺序一样。
  + 乱序流水线：流水线输出端任务的流出顺序与输入端的任务的流入顺序可以不同，允许后进入流水线的任务先执行。
+ 按数据表示
  + 标量处理机
  + 向量处理机

### 流水线设计中的问题
+ 瓶颈问题：当流水线各段不均匀时，机器的时钟周期取决于瓶颈段的延迟时间。所以在设计流水线时，要尽可能使各段时间相等
+ 流水线的额外开销
  + 流水寄存器延迟：流水寄存器需要建立时间（在出发写操作的时钟信号到达之前，寄存器输入端数据必须保持稳定的时间）和传输延迟（时钟信号到达后，到寄存器输出端数据有效可用所需的时间）
  + 时钟偏移开销：流水线中时钟到达各流水寄存器的最大差值时间
+ 几个问题
  + 流水线并不能减少单条指令的执行时间，但却能提高吞吐率
  + 增加流水线深度可以提高流水线性能
  + 流水线深度受限于流水线额外开销
  + 当时钟周期已经小到与额外开销相同时，流水已没有意义。因为这时在每一个时钟周期中已没有时间来做有用工作

### 相关与冲突
#### RISC
+ 特征：
  + 单一指令长度（4B）
  + 较少的寻址长度（一般<=5）
  + 无存储器间接寻址
  + 每条指令最多有一个存储器操作数
  + LOAD/STORE指令不会与算术运算混在一起
  + 整数寄存器有32个以上
  + 浮点寄存器有16个以上

RISC的思想是减少CPI，代价是在相同的CPU时钟周期下指令条数更多

#### MIPS指令系统结构
1. 32个64位通用寄存器（整数寄存器）：R0值永远是0
2. 32个64位浮点寄存器
3. 一些特殊寄存器，如状态寄存器
4. 数据表示
  + 整数：字节（8位），字（32位）。字节、半字或者字在装入64位寄存器时使用0扩展或者符号位扩展
  + 浮点数：单精度32位，双精度64位
5. 寻址方式
   + 立即数寻址与偏移量寻址，立即数子段和偏移量子段都是16位的
   + 寄存器间接寻址是把0作为偏移量实现的
   + 16位绝对寻址是通过把R0作为基址寄存器来完成的
   + MIPS的存储器是按字节寻址的，地址为64位
   + 所有存储器访问都是边界对齐的

#### 流水线冲突和相关
+ 相关：两条指令之间存在某种依赖关系，如果两条指令相关，那么它们就有可能不能在流水线中重叠执行或者只能部分重叠执行
  + 数据相关：如果两条指令i在前，j在后，如果指令j使用i产生的结果或者指令j与k数据相关，k与i数据相关（传递性），则说i与j数据相关。数据相关反应了数据的流动关系。
  + 名（指令所访问的寄存器或存储器单元的名称）相关：如果两条指令使用相同的名，但是他们之间没有数据流动，则称这两条指令存在名相关。名相关的两条指令之间没有数据的流动，如果一条指令的名改变了不影响另一条指令的执行。
    + 反相关：如果指令j写的名和指令i读的名相同，则称i和j反相关
    + 输出相关：指令j写的名=指令i写的名

#### 计分牌算法
![alt text](/system-struct/image-45.png)
**流程步骤**
+ 流出
  + 如果当前流出指令所需功能部件全部空闲，并且所有其他正在执行的指令的目的寄存器与该指令不同，计分牌就向功能部件流出该指令，并修改计分牌内部的记录表（解决了WAW）
+ 读操作数
  + 计分牌监测源操作数的可用性，一旦数据可用，它就通知功能部件从寄存器读出源操作数并开始执行，否则等待
  + 数据可用：如果所有前面已流出且还在执行的指令都不对该寄存器进行写操作，则寄存器的数据是可用的（解决了RAW）
+ 执行
  + 取到操作数后，功能部件开始执行。当产出结果后就通知计分牌它已经完成执行
+ 写结果
  + 计分牌一旦知道执行部件完成了执行，就监测是否存在WAR冲突。如果不存在，或者原有的WAR冲突已经消失，计分牌就通知功能部件把结果写入目的寄存器，并释放该指令使用的所有资源；如果监测到冲突，就不允许该指令将结果写到目的寄存器，这发生在前面的某条指令还没有读取操作数，而且其某个源操作数寄存器与本指令的目的寄存器相同。在这种情况下，计分牌必须等待，直到冲突消失。

#### Tomasulo
**核心思想**
+ 记录和检测指令相关，操作数一旦就绪就立即执行，把RAW冲突的可能性减到最小
+ 通过寄存器换名来解决WAR，WAW冲突

**保留站**
+ 每个保留站保留一条已经流出并等待本功能部件执行的指令
+ 包括：操作码，操作数以及用于检测和解决冲突的信息
  + 在一条指令流出到保留站的时候，如果该指令的源操作数已经在寄存器中就绪，则将之取到保留站中
  + 如果操作数还没有计算出来，则在该保留站中记录将产生这个操作数的保留站的标识

**公共数据总线CDB**
+ 所有功能部件的计算结果都是送到CDB上，由他把这些结果直接送到各个需要改结果的地方
+ 在具有多个执行部件且采用多流出的流水线中，需要采用多条CDB

**load缓冲器和store缓冲器**
+ 存放读/写存储器的数据或地址
+ load缓冲器的作用有三个：
  + 存放用于计算有效地址的分量
  + 记录正在进行的load访存，等待存储器响应
  + 保存已经完成了的load的结果，等待CBD传输
+ store缓冲器的作用有三个
  + 存放计算有效地址的分量
  + 保存正在进行的store访存的目标地址，该store正在等待存储数据的到达
  + 保存改store的地址和数据，直到存储部件接收

**浮点寄存器FP**
+ 共有16个浮点寄存器
+ 他们通过一对总线连接到功能部件，并通过CDB连接到store缓冲器

**指令队列**
+ 指令部件送来的指令放入指令队列
+ 指令队列中的指令按先进先出的顺序流出

**运算部件**
+ 加法器，乘法器

+ 冲突检测逻辑是分布的
  + 如果有多条指令已经获得了一个操作数，并同时在等待同一运算结果，那么这个结果一产生，就可以通过CDB同时送给所有这些指令，让他们同时进行
+ 消除了WAW，WAR冲突导致的停顿

#### 动态分支预测技术
在程序运行时，根据分支指令过去的表现来预测将来的行为
**分支预测的有效性决定于：**
+ 预测的准确性
+ 预测正确和不正确两种情况下的分支开销
+ 决定分支开销的因素：
  + 流水线的结构
  + 预测的方法
  + 预测错误时的恢复策略

**目的**
+ 预测分支是否成功
+ 尽快找到分支目标地址，避免控制相关造成流水线停顿

**需要解决的问题**
+ 如何记录分支的历史信息
+ 如何根据这些信息来预测分支的去向

在预测错误时，要作废已经预取和分析的指令，恢复现场，并从另一条分支路径重新取指令
![alt text](/system-struct/image-54.png)

**采用分支历史表BHT**
用BHT来记录分支指令最近一次或几次的执行情况（成功或不成功），并据此进行预测。BHT可以跟分支指令一起存放在指令cache中，也可以用一个专门的硬件来实现。

采用两位二进制位来记录历史
![alt text](/system-struct/image-55.png)

**步骤**
+ 分支预测
  + 当分支指令到达ID时，根据从BHT读出的信息进行分支预测
  + 若预测正确，就继续处理后续的指令，指令流水线没有断流。否则，就要作废已经预取和分析的指令，恢复现场，并从另一条分支路径重新取指令
+ 状态修改

**BHT只有在以下情况才有用：**
+ BHT指对分支是否成功进行预测，不提供分支的目标地址。因此在判定分支是否成功所需的时间大雨确定分支目标地址所需的时间时才有效
+ 对于前述5段经典流水线：由于判定分支是否成功和计算分支目标地址都是在ID段完成，所以BHT方法不会给该流水线带来好处

**采用分支目标缓冲器BTB**
+ 对于高性能流水线，如多流出类的处理机，在准确预测分支的基础上，还要求能快速提供足够的指令流
+ 对于5段经典流水：如果能提前1拍，在IF端得到足够信息，就可能将分支的开销降为0
+ 方法：分支目标缓冲
  + 将分支成功的分支指令的地址和他分支目标地址都放到一个缓冲区中保存起来，缓冲区以分支指令的地址作为标识
  + 这个缓冲区就是BTB

#### 基于硬件的前瞻执行
+ 前瞻执行的基本思想（允许指令乱序执行，必须顺序确认）：
  + 对分支指令的结果进行猜测，并假设这个猜测总是对的
  + 然后按这个猜测结果继续取、流出和执行后续的指令
  + 只是执行指令的结果不是写回到寄存器或存储器，而是放到一个称为ROB的缓冲队列中。等到相应的指令的到确认之后，才将结果写入寄存器或存储器
+ 对Tomasulo算法加以补充，就可以支持前瞻执行
  + 把Tomasulo算法的写结果和指令完成加以区分，分成两个不同段：写结果，指令确认
+ 写结果段
  + 把前瞻执行的结果写到ROB中
  + 通过CDB在指令之间传送结果，供需要用到这些结果的指令使用
+ 指令确认段
  + 在分支指令的结果出来后，对相应指令的前瞻执行给予确认
  + 如果前面所做的猜测是对的，把在ROB中的结果写到寄存器或存储器。
  + 如果发现是错的，那就不予确认，并从那条分支指令的另一条路径开始重新执行

### 消息传递机制
当源节点和目的节点之间没有直接的连接时，消息需要经过中间的节点进行传递。寻径就是用来实现这种传递的通信方法和算法。有的称之为路由
![alt text](/system-struct/image-89.png)

+ 寻径方式
  + 线路交换：在传递一个信息之前，需要先建立一条从源节点到目的节点的物理通路，然后再传递信息
    + 优点：传输带宽大，平均传输时延小，而且使用的缓冲区小
    + 缺点：需要频繁的建立源节点到目的节点到物理通路，时间开销大
  + 存储转发（分组交换）
    + 所经过每个中间节点都要设置一个包缓冲器，用于保存所传递的包。当一个包到达某中间节点时，该节点先把这个包全部存储起来，然后在出口链路可用、而且下一个节点的包缓冲器也可用的情况下，传递给下一个节点
  + 虚拟直通
    + 没有必要等到信息包全部放入缓冲器后再作路由选择，只要接收到用作寻径的包头就可以做出判断
    + 当出现寻径阻塞时，虚拟直筒方式需要将整个信息包全部存储在寻径节点中，要求每个节点都有足够大的缓冲区
  + 虫蚀方式：把信息包“切割”成更小的单位“片”，而且使信息包中各片的传送按流水方式进行
    ![alt text](/system-struct/image-90.png)
    ![alt text](/system-struct/image-91.png)

**确定性寻径**：通信路径完全由源节点地址和目的地址来决定，也就是说，寻径路径时预先唯一确定好的，与网络状况无关
**自适应寻址**：通信的通路每一次都要根据资源或者网络的情况来选择。可以避开拥挤的或者有故障的节点，使网络的利用率得到改进

## 第四章 向量处理机

### 向量处理方式
+ 横向处理方式
  + 向量计算是按行的方式从左到右横向的进行
    + 先计算：$d_1 \leftarrow a_1*(b_1+c_1)$
    + 再计算：$d_2 \leftarrow a_2*(b_2+c_2)$
  + 组成循环程序进行处理
    + $k_i \leftarrow b_i+c_i$
    + $d_i \leftarrow k_i*a_i$
    + 数据相关：N次   功能切换：2N次
  + 不适合于向量处理机的并行处理
+ 纵向处理方式
  + 向量计算是按列的方式从上到下纵向的进行
    + 先计算：$k_1 \leftarrow b_1+c_1...k_n$
    + 再计算：$d_1 \leftarrow k_1*a_1...d_n$
  + 表示成向量指令：
    + K=B+C   D=K*A
    + 两条向量指令之间：数据相关：1次；功能切换：1次
  + 向量处理机方式
+ 纵横（分组）处理方式
  + 把向量分成若干组，组内按纵向方式处理，依次处理各组

### 向量处理机及其系统结构

**进行向量链接的要求**
+ 无向量寄存器使用冲突和无功能部件使用冲突
+ 只有在前一条指令的第一个结果元素送入结果向量寄存器的那一个时钟周期才可以进行链接

## 第五章 多处理机
根据存储器的组织结构，将现有的MIMD机器分为两类：
+ 集中式共享存储器结构
  + 各处理器共享一个集中式的物理存储器，这类机器又是被称为对称式共享存储器多处理机（SMP）
  ![alt text](/system-struct/image-105.png)
+ 分布式存储器结构
  + 存储器在物理上是分布的
  + 每个节点包括：处理器，存储器，I/O，互联网络接口
  ![alt text](/system-struct/image-106.png)
  + 优点：如果大多数的访问是针对本节点的局部存储器，则可降低对存储器和互联网络的带宽要求；对本地存储器访问的延迟小
  + 缺点：处理器之间的通信较为复杂，且各处理器之间访问延迟较大
  + 簇：超级节点
    + 每个超级节点内包含个数较少的处理器
    + 超级节点之间通过分布式互联网络进行连接
    + 节点内部，处理器之间的连接可采用另一种互联技术，如总线
+ 两种存储器系统结构（分布式共享存储器）
  + 物理上分离的所有存储器共享统一的逻辑空间进行编址
  + 任何一个处理器可以访问该共享空间的任何一个单元，而且不同处理器上的同一个物理地址指的是同一个存储单元
  ![alt text](/system-struct/image-107.png)

### cache一致性问题
+ 允许共享数据进入cache，就可能出现多个处理器的cache中都有同以存储块的副本
+ 当其中某个处理器对其cache中的数据进行修改后，就会使得其cache中的数据与其他cache中的数据不一致
+ 例如：共享可写的数据、进程迁移、I/O传输
![alt text](/system-struct/image-113.png)
![alt text](/system-struct/image-114.png)
![alt text](/system-struct/image-115.png)
+ 一致性：如果对某个数据项的任何读操作均可得到其最新写入的值，则认为这个存储系统是一致的
![alt text](/system-struct/image-116.png)

### 实现一致性的基本方案
在支持cache一致性的多处理机中，cache提供两种功能：
  + 共享数据的迁移-将原称的共享数据拷贝迁入本地cache
  + 共享数据的复制-把多个处理器需要同时读取的共享数据在这些处理器本地cache中各存一个版本

### 监听协议
![alt text](/system-struct/image-117.png)
利用总线具有的广播能力，用分散控制的办法解决cache一致性问题
+ 写无效策略：在本地cache的数据块修改时使远程数据块都无效
+ 写更新策略：在本地cache数据块修改时通过总线把新的数据块广播给含该数据块的所有其他cache
+ 差别
  + 在对同一个数据进行多次写操作而中间无读操作的情况下，写更新协议需进行多次写广播操作，而写作废协议只需一次作废操作
  + 写作废是针对cache块进行操作，而写更新则是针对字或字节进行
  + 考虑从一个处理器A写操作后到另一个处理器B能读到该写入数据之间的延迟时间，写更新协议的延迟时间较小

**数据块三种状态**
+ 无效（I）：cache中该块的内容为无效
+ 共享（S）：该块可能处于共享状态。在多个处理器中都有副本，这些副本都相同，且与存储器中相应的块相同
+ 已修改（M）：该块已经被修改过，并且还没有写入存储器。块中的内容是最新的，系统中唯一的最新副本

### 目录协议
+ 分布式目录： 目录与存储器一起分布到各节点中，从而对于不同目录内容的访问可以在不同的节点进行
![alt text](/system-struct/image-124.png)
每个目录项包含1个位向量：记录哪些cache中有副本。由位向量制定的处理机的集合称为共享集S
+ 最简单的实现方案：对于存储器中每一块都在目录中设置一项。目录中的信息量与M*N成正比
  + M：存储器中存储块的总数量；N：处理器的个数
  + 由于M=K*N,K是每个处理机中存储块的数量，所以如果K保持不变，则目录中的信息量与$N^2$成正比
+ 存储块状态
  + **未缓冲**：该块尚未被调入cache，所有处理机的cache中都没有这个块的副本
  + **共享**：该块在一个或多个处理机上有这个块的副本，且这些副本与存储器中的该块相同
  + **独占**：仅有一个处理机有这个块的副本，且该处理机已经对其进行了写操作，所以其内容是最新的，而存储器中该块的数据已经过时，这个处理机称为该块的拥有者
+ 结点类型
  + 宿主节点：包含访问的存储单元及其目录项的节点
  + 本地节点：发出访问请求的节点
  + 远程节点：可以和宿主节点是同一个节点，也可以不是

+ 目录结构分类

    ![alt text](/system-struct/image-133.png)
  + 有限映像目录
    + 采用位数固定的目录项目，限制同一数据块在所有cache中的副本总数。
    ![alt text](/system-struct/image-134.png)
  + 链式映像目录
    + 用一个目录指针来表示共享集合。当一个数据块的副本数增加时，其指针链表就跟着变长
    + 由于链表的长度不受限制，其优点是：既不限制副本个数，又保证了可扩展性
